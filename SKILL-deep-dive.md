# Prompt Deep Dive Skill

주간 회고가 "이번 주 전체의 변화"를 추적한다면, 이 스킬은 "이 세션이 왜 이렇게 됐는가"를 파고든다.

<!-- include:LOG-SPEC -->

## 진입 방식

이 스킬은 사용자가 분석 대상을 지정하는 것이 기본이다. 자동으로 세션을 골라서 전부 분석하지 않는다.

### A. 주간 회고에서 드릴다운

주간 회고 결과를 본 후 특정 세션을 지정한다:
- "비효율 세션 1번 자세히 봐줘"
- "3월 25일 stradv-mlops 세션 분석해줘"
- "턴 수 가장 많았던 세션 파고들어줘"

이 경우 주간 회고에서 이미 로그를 읽은 상태이므로, 해당 세션의 session_id나 날짜+프로젝트 조합으로 특정한다.

### B. 방금 끝난 세션 즉시 복기

세션이 삽질이었다고 느꼈을 때 바로 분석한다:
- "방금 세션 분석해줘"
- "지금 끝난 거 복기"

이 경우 오늘 날짜의 가장 최근 로그 항목을 분석 대상으로 한다:

```bash
# 오늘 로그의 마지막 세션
tail -1 ~/.claude/session-logs/$(date +%Y-%m-%d).jsonl
```

### C. 대상이 불분명한 경우

"프롬프트 분석해줘"처럼 대상을 지정하지 않았을 때:
1. 먼저 사용자에게 어떤 세션을 분석하고 싶은지 물어본다.
2. 참고할 수 있도록 최근 세션 목록을 간략히 보여준다:

```bash
# 최근 5개 세션 요약
tail -5 ~/.claude/session-logs/$(date +%Y-%m-%d).jsonl | \
  python3 -c "
import json, sys
for line in sys.stdin:
    s = json.loads(line.strip())
    print(f\"  [{s['elapsed_human']}] {s['turn_count']}턴 | {s['cwd'].split('/')[-1]} | {s['prompts'][0][:50]}...\")
"
```

3. 사용자가 선택하면 해당 세션을 분석한다.

## 분석 절차

대상 세션이 확정되면 아래 순서로 분석한다.

### 1단계: 세션 개요

| 항목 | 값 |
|------|-----|
| 날짜 | ... |
| 프로젝트 | (cwd의 마지막 2 경로) |
| 소요 시간 | elapsed_human |
| 턴 수 | turn_count |
| 도구 사용 | tool_usage (도구별 횟수) |

### 2단계: 턴별 흐름 추적

prompts 배열의 각 항목을 순서대로 나열하고 흐름을 분석한다.

각 턴에 대해:
- 이 턴에서 사용자가 의도한 것은 무엇인가
- 이전 턴과의 관계: 새로운 요청인가, 보정인가, 추가 조건인가
- 여기서 방향이 꼬였다면 원인은 무엇인가

출력 형식:

```
턴 1: "에러 나는데 고쳐줘"
  -> 의도: 에러 수정 요청
  -> 문제: 어떤 에러인지, 어느 파일인지 정보 없음

턴 2: "이것도 안 돼"
  -> 의도: 1턴 결과가 기대와 다름
  -> 문제: "이것"이 무엇인지, "안 된다"의 증상이 무엇인지 불명확

턴 3: "아 그게 아니라 pods가 pending인 거야"
  -> 의도: 맥락 보정 (처음부터 이 정보를 줬어야 했음)
  -> 전환점: 여기서 비로소 실질적 디버깅 시작
```

### 3단계: 문제 진단

아래 유형 중 해당하는 것을 진단한다. 복수 해당 가능.

| 유형 | 징후 |
|------|------|
| 맥락 부족 | 첫 턴에 파일명, 에러 메시지, 기대 결과 없음 |
| 점진적 명세 | "아 그리고...", "근데 이것도..." 패턴으로 조건이 추가됨 |
| 범위 과다 | 한 세션에서 프롬프트 주제가 2회 이상 전환 |
| 출력 형식 미지정 | "이거 말고", "다시 해줘" 같은 수정 요청 반복 |
| 디버깅 루프 | 유사한 프롬프트가 연속으로 3회 이상 |
| 환경 문제 | 작업 자체가 아닌 셋업/설정에 턴을 소모 |

### 4단계: Before/After 제안

첫 턴(또는 방향이 꼬인 턴)의 원래 프롬프트를 개선한다.

규칙:
- Before는 실제 프롬프트 원문을 그대로 인용
- After는 이 세션에서 나중에 밝혀진 정보를 포함하여, 1턴에 끝낼 수 있도록 재작성
- 변경 포인트를 한 줄로 설명

```
Before:
> 에러 나는데 고쳐줘

After:
> stradv-mlops/deploy.yaml에서 pods가 Pending 상태야.
> kubectl describe pod 결과: 0/3 nodes are available: insufficient nvidia.com/gpu.
> GPU resource limits 설정이 빠진 것 같은데 확인하고 수정해줘.

변경 포인트: 3턴에 걸쳐 드러난 맥락(파일, 증상, 원인 추정)을 첫 턴에 모두 포함.
```

### 5단계: 구조적 해결 제안

이 세션의 문제가 "프롬프트를 더 잘 쓰면 해결"이 아니라 "자동화하면 해결"인 경우를 식별한다:

- 같은 유형의 작업이 매번 반복된다면 -> slash command 제안
- 프로젝트별 맥락이 매번 필요하다면 -> CLAUDE.md에 넣을 내용 제안
- 환경 설정 문제가 반복된다면 -> SessionStart hook 또는 setup script 제안

해당 사항이 없으면 이 단계는 생략한다.

## 리포트 형식

```
# 세션 분석: {프로젝트명} ({날짜})

## 개요
(1단계 테이블)

## 턴별 흐름
(2단계 분석)

## 진단: {주요 문제 유형}
(3단계 진단과 근거)

## Before/After
(4단계 제안)

## 구조적 해결 (해당 시)
(5단계 제안)
```

복수 세션을 분석하는 경우에만 마지막에 "공통 패턴" 섹션을 추가한다.

## 주의사항

- 프롬프트 원문 인용 시 API 키, 비밀번호 등 민감 정보는 `***`로 마스킹한다.
- "이 프롬프트가 나빴다"가 아니라 "이렇게 쓰면 더 빠르게 끝낼 수 있었다"로 표현한다.
- 턴 수가 많아도 탐색적 작업(기술 조사, 아키텍처 논의)은 비효율이 아니다. 작업 성격을 고려한다.
- 세션의 prompts 배열과 집계된 메타데이터(tool_usage, turn_count, elapsed_seconds)만으로 분석한다. raw transcript를 추가로 읽지 않는다. 이 때문에 "3턴째에서 Bash를 6번 호출하며 삽질"과 같은 턴별 도구 분포 진단은 불가능하다. 턴별 분석은 프롬프트 텍스트의 흐름에 집중하고, 도구 사용은 세션 전체 수준에서만 참고한다.
