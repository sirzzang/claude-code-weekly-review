# Prompt Deep Dive Skill

Claude Code 세션 로그에서 비정상적인 세션을 식별하고, 턴 단위로 흐름을 추적하여 프롬프트 개선 방안을 구체적으로 제시한다.

## 전제 조건

- Stop hook(`~/.claude/hooks/log-session.py`)이 설정되어 있어야 한다.
- 로그 파일은 `~/.claude/session-logs/YYYY-MM-DD.jsonl` 형식으로 존재한다.

## 로그 형식

weekly-review 스킬과 동일한 로그를 사용한다:

```json
{
  "timestamp": "ISO 8601",
  "date": "YYYY-MM-DD",
  "session_id": "string",
  "cwd": "작업 디렉토리",
  "prompts": ["사용자 프롬프트 목록"],
  "turn_count": 3,
  "tool_usage": {"Bash": 5, "Read": 3, "Edit": 2},
  "tool_total": 10,
  "elapsed_seconds": 180,
  "elapsed_human": "3m 0s"
}
```

## 실행 절차

### 1단계: 로그 수집

```bash
# 최근 7일간 로그 파일 확인
ls -la ~/.claude/session-logs/ | tail -10

# 최근 7일 로그를 합쳐서 읽기
find ~/.claude/session-logs/ -name "*.jsonl" -mtime -7 -exec cat {} + 2>/dev/null
```

사용자가 특정 기간을 지정하면 해당 기간만 필터링한다:

```bash
find ~/.claude/session-logs/ -name "*.jsonl" | sort | while read f; do
  date_part=$(basename "$f" .jsonl)
  if [[ "$date_part" >= "2026-03-17" && "$date_part" <= "2026-03-23" ]]; then
    cat "$f"
  fi
done
```

### 2단계: 비정상 세션 식별

아래 기준으로 비정상 세션 후보를 추출한다. 최대 5개를 선별한다.

| 기준 | 임계값 | 의미 |
|------|--------|------|
| 높은 턴 수 | 상위 20% 또는 turn_count >= 8 | 프롬프트가 불명확하여 반복했을 가능성 |
| 과도한 도구 사용 | tool_total이 평균의 2배 이상 | 범위가 너무 넓거나 삽질 |
| 높은 턴 대비 짧은 프롬프트 | 평균 프롬프트 길이 < 20자 && turn_count >= 4 | 맥락 없이 짧은 지시를 반복 |
| 긴 소요 시간 | elapsed_seconds 상위 20% | 비효율적 진행 |
| 높은 턴 대비 낮은 도구 사용 | turn_count >= 5 && tool_total <= 2 | 대화만 반복되고 실질 작업 없음 |

복수 기준에 해당하는 세션을 우선 선별한다.

### 3단계: 세션별 상세 분석

선별된 각 세션에 대해 아래 항목을 분석한다.

#### 3-1. 세션 개요

- session_id, 날짜, 프로젝트(cwd), 소요 시간, 턴 수, 도구 사용 요약

#### 3-2. 턴별 흐름 추적

각 턴의 프롬프트를 순서대로 나열하고, 흐름이 어디서 꼬였는지 식별한다:

- 1턴: 사용자가 무엇을 요청했는가
- 2턴 이후: 왜 추가 턴이 필요했는가 (불명확, 범위 변경, 오류 수정 등)
- 어느 턴에서 방향이 바뀌었는가

#### 3-3. 문제 진단

아래 유형 중 해당하는 것을 진단한다:

| 유형 | 설명 | 징후 |
|------|------|------|
| 맥락 부족 | 배경, 제약 조건, 기대 결과 누락 | "이거 고쳐줘" 류의 첫 프롬프트 |
| 범위 과다 | 한 세션에 여러 독립적 작업 | cwd 동일하지만 프롬프트 주제가 전환 |
| 점진적 명세 | 처음에 대충 시키고 후속 턴에서 조건 추가 | "아 그리고...", "근데 이것도..." |
| 출력 형식 미지정 | 원하는 형태를 안 알려주고 수정 요청 | "이거 말고...", "다시 해줘" |
| 디버깅 루프 | 같은 에러를 반복 시도 | 유사한 프롬프트가 연속 |

#### 3-4. Before/After 제안

원래 프롬프트(Before)와 개선된 버전(After)을 구체적으로 제시한다.

Before/After 작성 규칙:
- Before는 실제 프롬프트 원문을 그대로 인용한다
- After는 1턴에 끝낼 수 있도록 맥락, 범위, 형식을 포함하여 재작성한다
- 변경한 부분과 이유를 간단히 설명한다

### 4단계: 리포트 작성

아래 구조로 출력한다. 마크다운 형식을 사용한다.

```
# 프롬프트 상세 분석: {시작일} ~ {종료일}

## 분석 대상
(선별 기준과 총 세션 수 대비 비정상 세션 수)

## 세션 1: {세션 요약 한 줄}

### 개요
| 항목 | 값 |
|------|-----|
| 날짜 | ... |
| 프로젝트 | ... |
| 소요 시간 | ... |
| 턴 수 | ... |
| 도구 사용 | ... |

### 턴별 흐름
(각 턴의 프롬프트와 흐름 분석)

### 진단
(문제 유형과 근거)

### Before/After
**Before:**
> 원래 프롬프트

**After:**
> 개선된 프롬프트

**변경 이유:** ...

---

## 세션 2: {세션 요약 한 줄}
(동일 구조 반복)

---

## 공통 패턴
(여러 세션에서 반복되는 문제가 있으면 정리)

## 핵심 개선 포인트
(실행 가능한 1-3개 액션 아이템)
```

## 주의사항

- 프롬프트 원문에 민감 정보(API 키, 비밀번호 등)가 포함될 수 있다. 인용 시 민감 정보는 `***`로 마스킹한다.
- 비판이 아닌 개선 제안의 톤을 유지한다. "이 프롬프트가 나쁘다"가 아니라 "이렇게 쓰면 더 빠르게 끝낼 수 있었다"로 표현한다.
- 데이터가 부족하면(세션 5개 미만) 있는 것만으로 분석하되, 데이터 부족을 명시한다.
- 비정상 세션이 없으면 "전반적으로 프롬프트 품질이 양호하다"고 판단하고, 가장 효율적이었던 세션을 좋은 사례로 제시한다.
