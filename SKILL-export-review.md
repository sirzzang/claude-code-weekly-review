# Export Review Skill

weekly-review 또는 prompt-deep-dive 스킬의 리포트를 로컬 파일 또는 외부 서비스에 저장한다.

## 실행 절차

### 1단계: 리뷰 내용 확인

현재 대화에서 가장 최근의 리뷰 리포트를 식별한다.

식별 기준:
- `# 주간 회고:` 로 시작하는 블록 → weekly-review 리포트
- `# 세션 분석:` 로 시작하는 블록 → deep-dive 리포트

리포트가 없으면:
```
리뷰 리포트를 찾을 수 없습니다.
먼저 /weekly-review 또는 /prompt-deep-dive를 실행한 후 다시 시도해 주세요.
```

### 2단계: 저장 대상 결정

#### 2-1. 사용자가 대상을 지정한 경우

"노션에 올려줘", "로컬에만 저장해줘" 등 대상을 명시한 경우 해당 서비스만 실행한다. 감지된 전체 목록을 나열하지 않는다.

#### 2-2. 대상을 지정하지 않은 경우

사용 가능한 외부 서비스를 감지하여 선택지를 제시한다.

**MCP 감지 방법:**

현재 대화에서 사용 가능한 도구 목록을 확인하고 아래 핸들러 레지스트리와 매칭한다. `mcp__notion__API-post-page` 등 해당 접두사의 도구가 하나라도 있으면 해당 서비스를 "사용 가능"으로 판단한다.

**선택지 제시 형식:**

MCP가 감지된 경우:
```
리뷰 결과를 어디에 저장할까요?
1) 로컬 파일만 (~/.claude/review-reports/)
2) 로컬 파일 + Notion
3) 로컬 파일 + Confluence
(번호 또는 서비스명으로 선택하세요. 복수 선택 가능)
```

MCP가 감지되지 않은 경우:
```
리뷰 결과를 로컬 파일에 저장합니다.
경로: ~/.claude/review-reports/

외부 서비스(Notion, Confluence 등)에도 업로드하려면
해당 MCP 서버를 연결한 후 다시 시도해 주세요.
```

**핵심 원칙:**
- 외부 업로드는 사용자가 명시적으로 선택한 서비스에만 실행
- MCP가 여러 개 감지되어도 자동 실행하지 않음
- 로컬 파일 저장은 기본값 (사용자가 "로컬은 빼줘"라고 하지 않는 한 항상 실행)

### 3단계: 로컬 파일 저장

#### 저장 경로

```
~/.claude/review-reports/
├── 2026-03-29-weekly-review.md
├── 2026-03-30-deep-dive-abc123.md
└── ...
```

#### 파일명 규칙

| 리뷰 타입 | 파일명 패턴 |
|-----------|------------|
| weekly-review | `YYYY-MM-DD-weekly-review.md` (리뷰 실행일 기준) |
| deep-dive | `YYYY-MM-DD-deep-dive-{session_id 앞 6자}.md` |

같은 날 같은 타입의 리포트가 이미 존재하면 `-2`, `-3` 접미사를 붙인다:
- `2026-03-29-weekly-review.md`
- `2026-03-29-weekly-review-2.md`

#### 파일 구조

```markdown
---
date: 2026-03-29
type: weekly-review
period: 2026-03-23 ~ 2026-03-29
---

(리뷰 리포트 마크다운 원문)
```

```markdown
---
date: 2026-03-30
type: deep-dive
session_id: abc123
session_date: 2026-03-28
project: my-project
---

(리뷰 리포트 마크다운 원문)
```

프론트매터의 메타데이터는 리포트 본문에서 추출한다:
- `period`: 주간 회고 제목의 날짜 범위
- `session_id`, `session_date`: 세션 분석 개요 테이블
- `project`: 세션 로그의 `cwd` 필드에서 마지막 2 경로 컴포넌트를 추출한 값 (별도 ID 체계가 아니라 동일 디렉토리에서 작업하면 항상 같은 값이 나온다)

#### 저장 실행

```bash
# 디렉토리 확인/생성
mkdir -p ~/.claude/review-reports

# 파일 쓰기 (Write 도구 사용)
```

저장 후 안내:
```
로컬 저장 완료: ~/.claude/review-reports/2026-03-29-weekly-review.md
현재 저장된 리포트: N개
```

### 4단계: 외부 서비스 업로드 (선택)

사용자가 외부 서비스를 선택한 경우에만 실행한다.

#### 업로드 핸들러 레지스트리

각 핸들러는 감지 조건, 업로드 방법, 속성 매핑을 정의한다. 새 서비스를 지원하려면 이 섹션에 핸들러를 추가한다.

---

### Notion 핸들러

**감지 조건:** `mcp__notion__` 접두사 도구가 사용 가능

**업로드 대상 레지스트리:**

사용자가 아래 레지스트리에 Notion DB/페이지를 등록한다. 등록된 대상이 없으면 사용자에게 DB ID 또는 페이지 URL을 직접 입력받는다.

<!-- 아래에 Notion 업로드 대상을 등록한다. -->
<!-- 예시:
| 이름 | 유형 | ID | 용도 |
|------|------|-----|------|
| 주간 회고 DB | database | xxxxxxxx-xxxx-xxxx-xxxx | 주간 회고 리포트 저장 |
| 학습 노트 | page | xxxxxxxx-xxxx-xxxx-xxxx | 학습 관련 리뷰 아카이빙 |
-->

**DB 업로드 시 속성 매핑:**

| 속성 | 규칙 |
|------|------|
| 이름/제목 | 리포트 제목 (예: "주간 회고: 2026-03-23 ~ 2026-03-29") |
| 날짜 | 리뷰 실행일 |
| 유형 | weekly-review 또는 deep-dive |
| 요약 | 핵심 지표 변화 또는 진단 결과 1~2줄 |

기존 DB의 속성 스키마를 먼저 조회하고, 매칭 가능한 속성만 채운다. 새 속성을 만들지 않는다.

**페이지 업로드 시:**
- 지정된 페이지의 하위 페이지로 생성
- 페이지 제목: 리포트 제목
- 본문: 리포트 마크다운을 Notion 블록으로 변환

**업로드 절차:**

1. 레지스트리에 등록된 대상이 있으면 선택지 제시, 없으면 직접 입력 요청
2. DB인 경우 `mcp__notion__API-retrieve-a-database`로 스키마 조회
3. `mcp__notion__API-post-page`로 페이지 생성
4. 생성된 페이지 URL 안내

---

### Confluence 핸들러

**감지 조건:** `mcp__confluence__` 또는 `atlassian` 관련 MCP 도구가 사용 가능

**업로드 절차:**

1. 사용자에게 업로드할 Space와 상위 페이지 확인
2. 해당 API로 페이지 생성
3. 생성된 페이지 URL 안내

---

### GitHub 핸들러

**감지 조건:** `gh` CLI가 사용 가능하고 인증된 상태

**업로드 절차:**

1. 사용자에게 리포지토리와 유형(Issue/Discussion) 확인
2. `gh issue create` 또는 `gh api`로 Discussion 생성
3. 생성된 URL 안내

---

### 기타 서비스 확장

새 서비스를 추가하려면 이 레지스트리에 아래 형식으로 핸들러를 추가한다:

```
### {서비스명} 핸들러

**감지 조건:** {MCP 도구 접두사 또는 CLI 명령}
**업로드 절차:**
1. ...
2. ...
```

---

### 5단계: 완료 확인

모든 저장이 끝나면 결과를 요약한다:

```
저장 완료:
- 로컬: ~/.claude/review-reports/2026-03-29-weekly-review.md
- Notion: https://notion.so/... (페이지 제목)

저장된 리포트 총 N개 (weekly-review M개, deep-dive K개)
```

## 주의사항

- 리포트 원문에 민감 정보가 포함될 수 있다. 외부 서비스 업로드 전 민감 정보 여부를 확인한다.
- 본문 내 이모지 사용 금지.
- 외부 서비스에 새 속성(컬럼) 생성 금지. 기존 스키마에 맞춘다.
- 로컬 파일의 프론트매터 형식을 변경하지 않는다. 향후 자동 파싱에 사용될 수 있다.
